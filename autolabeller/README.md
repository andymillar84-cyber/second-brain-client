# second-brain-automation

Automation layer for the Google Keep → Notion "second brain". Currently one piece:
**keep-autolabel** — auto-labels new Keep notes on a schedule so the end-of-day flow is just
*verify → route*, not *label everything from scratch*.

This folder is a self-contained, clonable unit: `git clone` it onto a Mac, run `./install.sh`, done.

## How it works

An hourly **launchd** agent runs `autolabel.py`. Each run: lists `keep inbox` → finds notes with
**no topic label yet** → classifies them with `claude -p --model haiku` (uses the Claude subscription,
no API key) against the existing label vocabulary + `keep profiles` exemplars + keep-organizer
`learnings.md` → applies the chosen **topic** label and (if actionable) a **routing** label.

It is purely additive — it **never archives** and **never creates new labels**, so the downstream
`keep-organizer` / `keep-router` skills are untouched. It is **idempotent** (each note labelled once,
tracked in `~/.config/keep-autolabel/processed.json`).

### Why launchd, not a cloud routine / n8n
The job must read + write Keep, which only works through the local `keep` CLI + the Google **master
token** (`gkeepapi` — Keep has no real API). A cloud scheduler (Claude cloud routine, n8n cloud) can't
reach a local CLI/token, and can't execute code on this Mac. So the scheduler must be local. launchd
fires whenever the Mac is awake (broader than a local Claude routine, which only runs while the Claude
app is open). Full rationale: the ADR in the main workspace
(`active/second-brain-install/architecture-decision.md`).

### Where things run from
- **Code:** this folder (a home-level path — *not* under `~/Desktop`, which macOS TCC blocks launchd
  from reading).
- **State + logs:** `~/.config/keep-autolabel/`
- **LaunchAgent:** `~/Library/LaunchAgents/com.andrew.keep-autolabel.plist` (generated by `install.sh`).

### Failure alerts (so it can't die silently)
The job's most likely long-term failure is a **credential expiring** — the Google master token getting
revoked, or the `claude` login signing out. Those don't break on a restart (it auto-resumes on login);
they just make the hourly run error into the log, where no one would notice.

So a failed run posts a **macOS notification ONCE** (`osascript display notification`), then drops a
`~/.config/keep-autolabel/.failing` sentinel and stays quiet while the failure persists — no hourly
nagging. It pings once more (“recovered”) the next time a run succeeds and clears the sentinel. Net
effect: a dead token surfaces as a notification within the hour instead of rotting in `launchd.err.log`.
The end-of-day `keep-organizer` review is still the backstop — nothing is lost either way, notes just
sit unlabelled until then.

*(Notifications come from a LaunchAgent in your GUI session, so they appear — typically attributed to
“Script Editor”. If macOS has notifications muted for that, the failure still lands in the log.)*

## Prerequisites (already present on Andrew's Mac; the setup skill installs them on a client's)
- `keep` CLI at `~/.local/bin/keep` + a valid Google master token (`~/.config/keep-cli/token` or
  keep-mcp env in `~/.claude.json`).
- `claude` CLI at `~/.local/bin/claude`, signed in.
- System `python3` (stdlib only — no extra packages).

## Use

```bash
./install.sh                 # generate the LaunchAgent + load it (runs hourly, and once now)
./uninstall.sh               # remove the agent (keeps state/logs)
python3 autolabel.py --dry-run   # classify + log to the daily log, apply NOTHING
python3 autolabel.py             # run once, live
```

Check it / logs:

```bash
launchctl list | grep keep-autolabel
cat ~/.config/keep-autolabel/daily-$(date +%F).log     # your end-of-day review
tail ~/.config/keep-autolabel/launchd.err.log          # should be empty
```

## Gotchas
- **Mac asleep → polls skipped**, caught on next wake; the end-of-day `keep-organizer` run is the backstop.
- **Must live outside `~/Desktop`/`~/Documents`/`~/Downloads`** — macOS TCC blocks launchd from reading
  those; `python3` fails with `Operation not permitted`. A home-level folder like this is fine.
- **Label names contain `[ ]`** (e.g. `[01] Personal`) — JSON parsing is string-aware, not bracket-counting.
- **First live run re-labels the current inbox backlog** (expected). `--dry-run` first to preview.
- **Client clone:** the label vocabulary is read live from the client's own Keep, so the script is
  already client-agnostic — only the master token + the client's labels differ.
